# Vmq Node.js + Vue 3 + PostgreSQL 重构需求文档

## 1. 文档目标

本文档用于定义对 [szvone/Vmq](https://github.com/szvone/Vmq) 服务端与 Web 管理后台的现代化重构需求。重构目标是：

- 将原项目的 Java Spring Boot + H2 + 传统静态页面实现，重构为：
  - 后端：Node.js + Express + RESTful API
  - 前端：Vue 3 + Vite
  - 数据库：PostgreSQL
- 保持原有业务逻辑、接口语义、订单状态流转、监控端对接协议、后台操作能力不变
- 在不改变业务结果的前提下，提升代码可维护性、模块边界清晰度、可测试性、部署标准化能力

本次范围仅包含：

- 服务端接口
- Web 管理后台
- 支付页
- 数据存储与初始化
- 与现有安卓监控端的接口兼容

本次不包含：

- 安卓监控端 APK 的实现重构
- 新增支付业务能力
- 改变签名算法、订单规则、回调字段、订单状态定义
- 改变已有对外 URL 语义

## 2. 原项目业务概览

原项目本质上是一个“个人开发者免签收款中转系统”，通过安卓监控端监听微信/支付宝收款通知，再由服务端完成订单匹配、状态更新、异步通知与同步跳转。

核心链路如下：

1. 商户调用 `createOrder` 创建订单
2. 服务端根据金额和支付方式分配实际支付金额 `reallyPrice`
3. 用户在支付页扫码付款
4. 安卓监控端监听到付款通知后调用 `appPush`
5. 服务端依据 `type + reallyPrice + state=0` 匹配待支付订单
6. 匹配成功后订单置为已支付，并向商户异步回调 `notifyUrl`
7. 前端支付页轮询 `checkOrder`，支付成功后跳转 `returnUrl`
8. 定时任务关闭超时未支付订单，并释放金额占位

## 6. 现有核心逻辑拆解

## 6.1 订单创建逻辑

- 校验 `payId`、`type`、`price`、`sign`
- 签名规则保持为：`md5(payId + param + type + price + key)`
- 生成平台订单号 `orderId`
- 根据 `payQf` 决定金额递增或递减
- 通过临时金额占位表避免相同支付方式下金额冲突
- 若存在固定金额二维码，则优先使用固定二维码，否则使用通用二维码
- 保存订单并返回创建结果

## 6.2 金额占位逻辑

- 原系统使用 `tmp_price` 表保存 `type-reallyPrice`
- 插入成功表示金额未被占用
- 插入失败则按 `0.01` 步进继续试探
- 订单关闭、支付成功、手动删除时释放占位

这是重构时必须重点保真的逻辑。

## 6.3 收款推送逻辑

- 安卓监控端调用 `appPush(type, price, t, sign)`
- 校验时间偏差不超过 50 秒
- 校验签名：`md5(type + price + t + key)`
- 根据 `reallyPrice + type + state=0` 匹配订单
- 若未匹配到，记录为“无订单转账”
- 若匹配到：
  - 释放金额占位
  - 设置订单为已支付
  - 触发异步回调
  - 回调失败时状态标记为 `2`

## 6.4 支付状态查询逻辑

- `getOrder` 返回订单详情与支付页渲染所需数据
- `checkOrder` 用于支付页轮询
- 待支付：返回错误信息
- 已过期：返回错误信息
- 已支付：返回拼装好的同步跳转 URL

## 6.5 订单关闭与超时逻辑

- 商户可通过 `closeOrder` 主动关闭订单
- 定时任务按 `close` 配置关闭超时订单
- 关闭时必须释放金额占位

## 6.6 监控端状态逻辑

- `appHeart` 更新最后心跳时间与在线状态
- `getState` 返回：
  - `state`
  - `lastheart`
  - `lastpay`
- 定时任务若发现心跳超 60 秒，则将监控端状态置为离线

## 6.7 后台管理逻辑

- 登录校验
- 系统配置编辑
- 监控端配置查看
- 微信/支付宝二维码管理
- 订单列表查看与筛选

## 8. 模块需求明细

## 8.1 模块 A：认证与会话模块

### 目标

复刻当前管理员登录行为，并支撑后台接口鉴权。

### 功能要求

- 提供登录接口
- 使用当前配置表中的 `user/pass` 进行校验
- 登录成功后建立后台会话
- 所有后台接口均需登录态校验
- 提供退出登录能力

### 兼容要求

- 保持单管理员模型
- 不引入多角色权限体系

### 技术要求

- 可使用 Express Session 或签名 Cookie 方案
- 前后端分离后仍需确保后台接口鉴权简洁稳定

## 8.2 模块 B：系统设置模块

### 功能要求

- 获取全部系统设置
- 保存系统设置
- 初始化默认设置

### 关键配置项

- `user`
- `pass`
- `notifyUrl`
- `returnUrl`
- `key`
- `wxpay`
- `zfbpay`
- `close`
- `payQf`
- `lastheart`
- `lastpay`
- `jkstate`

### 兼容要求

- 保持配置项语义不变
- 初始化默认值与原系统一致：
  - 用户名 `admin`
  - 密码 `admin`
  - `close=5`
  - `payQf=1`
  - `jkstate=-1`

## 8.3 模块 C：订单模块

### 功能要求

- 创建订单
- 查询订单详情
- 查询支付状态
- 关闭订单
- 列表查询订单
- 删除订单
- 批量删除过期订单
- 删除七天前订单
- 补单

### 订单字段

- `id`
- `orderId`
- `payId`
- `createDate`
- `payDate`
- `closeDate`
- `param`
- `type`
- `price`
- `reallyPrice`
- `notifyUrl`
- `returnUrl`
- `state`
- `isAuto`
- `payUrl`

### 状态定义

- `-1`：订单过期
- `0`：待支付
- `1`：已支付
- `2`：已支付但异步通知失败

### 兼容要求

- `createOrder`、`getOrder`、`checkOrder`、`closeOrder` 外部语义不变
- 同步跳转拼接参数不变
- 异步回调参数与签名规则不变

## 8.4 模块 D：金额占位模块

### 功能要求

- 记录待占用金额
- 检测金额冲突
- 按规则步进出唯一 `reallyPrice`
- 在订单完成/关闭/删除/过期时释放金额

### 兼容要求

- 延续“同支付方式 + 同金额不可重复占位”的业务规则
- 保持 `0.01` 为步进粒度
- 保持 `payQf=1` 递增、`payQf=2` 递减语义

### 设计要求

- PostgreSQL 中建议显式建表保存占位记录
- 必须通过事务和唯一约束保证并发安全

## 8.5 模块 E：二维码管理模块

### 功能要求

- 新增固定金额二维码
- 删除固定金额二维码
- 列表查看固定金额二维码
- 区分微信/支付宝二维码
- 支持通用二维码配置与固定二维码并存

### 数据字段

- `id`
- `payUrl`
- `price`
- `type`

### 兼容要求

- 创建订单时优先匹配固定金额二维码
- 匹配到固定二维码时 `isAuto=0`
- 未匹配到时走通用二维码，`isAuto=1`

## 8.6 模块 F：监控端兼容模块

### 功能要求

- `appHeart`
- `appPush`
- `getState`
- 监控配置二维码生成

### 协议兼容要求

- 参数名保持不变
- 签名规则保持不变
- 时间偏差校验保持不变
- 返回字段结构保持可兼容

### 特殊要求

- 必须兼容现有安卓监控端，不要求升级 APK
- 后端路由路径不能随意调整

## 8.7 模块 G：回调与通知模块

### 功能要求

- 订单支付成功后向商户异步通知
- `notifyUrl` 优先使用订单级配置，否则使用系统级配置
- 返回值仅在响应体严格等于 `success` 时视为成功
- 失败时将订单状态改为 `2`
- 补单同样触发回调

### 回调参数

- `payId`
- `param`
- `type`
- `price`
- `reallyPrice`
- `sign`

### 签名规则

- `md5(payId + param + type + price + reallyPrice + key)`

## 8.9 模块 I：后台统计模块

  - 今日收款金额

## 8.10 模块 J：定时任务模块

### 兼容要求

## 8.11 模块 K：支付页模块

## 8.12 模块 L：管理后台界面模块

- 微信二维码管理页

## 16. 本次文档输出结论

本项目适合拆为“兼容层优先”的保守重构工程：

- 先保证旧接口和旧业务语义原样落地
- 再在新技术栈中重建内部模块边界
- 最后再考虑清理兼容层与增强能力

这能最大程度降低“逻辑不变”要求下的重构风险。
