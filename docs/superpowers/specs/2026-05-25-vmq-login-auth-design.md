# TP-14 前端登录与鉴权页设计说明

## 背景

`TP-14` 的目标是在 `TP-13` 已完成的前端后台框架基础上，落地真实可用的前端登录页与鉴权流程。

当前已知前提如下：

- `TP-13` 已完成后台壳子、受保护路由、登录页占位结构与基础 `auth store`
- 后端 `TP-04` 已存在 `/login` 与 `/logout` 接口，并使用 `express-session` 维护后台会话
- 当前 session 策略保持现状，不额外修改时长与存储方案
- 本次采用“前端调用真实登录接口，依赖后端 session”的实现路径

## 目标

- 将占位登录页替换为真实后台登录表单
- 让前端鉴权从本地 token 占位方案切换为基于后端 session 的真实状态同步方案
- 保证未登录访问后台页会跳转登录页
- 保证已登录状态下访问登录页会自动跳回后台首页
- 让退出登录、刷新页面、会话失效等场景都能正确同步前端状态

## 不在本次范围内

- 修改后端 session 过期时间
- 引入 token、JWT 或刷新令牌方案
- 修改后端登录规则
- 增加找回密码、验证码、多因素认证等扩展能力
- 重做后台整体视觉系统

## 设计方向

### 认证模型

本次采用“后端 session 为准，前端只同步状态”的方案：

- 前端登录时调用后端 `/login`
- 登录成功后，由浏览器自动保存 session cookie
- 前端本地只维护 `loggedIn`、`loading`、`errorMessage` 等界面状态
- 前端不再自己生成或长期保存 token

这样可以与当前后端实现保持一致，也能避免前后端出现双重鉴权模型。

### 登录页定位

登录页是后台入口页，不是营销页，因此保持简洁、专业和高可读性：

- 延续 `TP-13` 已建立的现代化 SaaS 风格
- 使用正式表单替换占位按钮
- 只保留必要信息：标题、说明、账号、密码、提交按钮、错误提示
- 避免加入无关装饰和复杂动画

## 信息架构

### 路由规则

- `/login`
  - 未登录时允许访问
  - 已登录时自动跳转 `/dashboard`
- `/dashboard` 及其他后台路由
  - 未登录时统一跳转 `/login`
  - 已登录时正常访问

### 状态来源

前端登录态不再以本地 token 作为真实来源，而是通过以下方式同步：

- 登录成功后，前端显式设置为已登录
- 退出登录后，前端显式设置为未登录
- 应用初始化时，前端调用会话探测逻辑确认当前 session 是否有效
- 收到接口 `401` 时，前端回收登录态并跳回登录页

## 技术架构

### auth store 角色调整

`auth store` 从“本地 token 占位仓库”升级为“会话状态同步仓库”，建议至少包含：

- `loggedIn`
- `loading`
- `errorMessage`
- `bootstrapped`

以及以下动作：

- `bootstrap()`
- `login(credentials)`
- `logout()`
- `markLoggedOut(message?)`
- `syncSession()`

### 请求层职责

请求层增加认证相关封装，负责：

- 调用 `/login`
- 调用 `/logout`
- 在需要时发送带凭证请求
- 统一处理 `401`
- 将会话失效信号回传给 `auth store`

### 会话探测策略

由于当前后端没有独立的“当前用户信息”接口，前端需要一个会话探测方式来确认 session 是否还有效。

本次建议优先采用：

- 在应用初始化时，请求一个受保护后台接口进行轻量探测

推荐探测目标：

- `/admin/session`

探测成功：

- 认为当前 session 仍有效

探测失败且返回 `401`：

- 认为当前未登录或会话已失效

此方式不需要额外修改后端接口协议，符合本次边界。

## 登录页交互设计

### 表单字段

- `user`
- `pass`

输入项应具备：

- 中文标签
- 合理 placeholder
- 基础必填校验

### 提交态

点击登录后：

- 按钮进入 loading 状态
- 输入框与按钮临时禁用
- 防止重复提交

### 成功态

登录成功后：

- 清空错误信息
- 更新 `auth store`
- 跳转 `/dashboard`

### 失败态

登录失败后：

- 不跳转
- 不清空账号输入
- 使用表单内联提示展示错误信息
- 输入区域保留轻量错误态

失败提示优先使用后端返回文案；如无明确文案，则回退为通用中文提示。

## 退出与失效处理

### 主动退出

用户点击退出后：

- 调用 `/logout`
- 前端立即清理登录态
- 跳转到 `/login`

本次范围内要求将退出入口一起做完整，不保留 `TP-13` 中的占位“清理登录态”按钮行为。

### 被动失效

当后台接口返回 `401` 时：

- 统一清理前端登录态
- 回到 `/login`
- 提示“当前会话已失效，请重新登录”

这样可以覆盖服务重启、浏览器会话结束或其他导致 session 丢失的情况。

本次预计主要涉及以下文件：

  - 会话探测辅助方法
  - 针对登录态与守卫的新测试

## 测试策略

### 单元测试

优先覆盖可纯逻辑测试的部分：

- 登录态状态切换
- 路由守卫判断
- 登录失败信息映射
- 会话失效后的清理逻辑

### 集成验证

至少验证以下场景：

- 正确账号密码可以登录
- 错误账号密码返回内联错误提示
- 未登录访问后台页会跳回登录页
- 已登录访问登录页会自动回到后台首页
- 退出登录后再次访问后台页会被拦截

## 验收标准

满足以下条件即可认为 `TP-14` 完成：

1. 管理员输入正确账号密码后可以进入后台
2. 错误账号密码会在表单内联提示，不误跳转
3. 未登录访问后台页会被拦截到 `/login`
4. 已登录访问 `/login` 会自动回到 `/dashboard`
5. 退出登录后会返回登录页，后台页再次受保护
6. 登录页视觉风格与 `TP-13` 后台框架保持一致

## 风险与应对

### 风险：前端登录态与后端 session 状态不一致

应对：

- 以后端 session 为最终依据
- 初始化时增加会话探测
- `401` 时统一回收状态

### 风险：登录失败提示不够清晰

应对：

- 使用表单内联提示
- 尽量直出后端返回错误
- 保留用户已输入账号

### 风险：实现时不小心继续依赖旧 token 逻辑

应对：

- 在 `auth store` 中明确移除 token 作为真实鉴权来源
- 守卫统一改为读取真实登录态

## 当前假设

- 后端 `/login` 与 `/logout` 接口行为保持不变
- 后端 `/admin/session` 可作为当前 session 的探测接口
- 浏览器会自动携带并保存 session cookie
- 当前阶段允许通过请求已有受保护接口来完成会话探测

## 结论

本设计将 `TP-14` 收敛为“真实登录表单 + 基于后端 session 的前端鉴权闭环”：不改后端会话期限，不引入新鉴权模型，只把前端从占位态升级为真实可用的后台登录与登录态同步机制。
