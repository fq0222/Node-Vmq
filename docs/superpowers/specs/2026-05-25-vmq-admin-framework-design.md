# TP-13 前端后台框架设计说明

## 背景

`TP-13` 的目标是为 Vue 3 Web 端建立可复用的后台管理壳子。根据 `docs/plans/2026-05-22-vmq-development-task-breakdown.md` 的任务拆解，这一阶段应聚焦“框架能力”，而不是具体业务实现。

当前已经确认的约束如下：

- 视觉方向：现代化 SaaS 仪表盘风格
- 交付边界：本次只完成 `TP-13` 后台壳子
- 当前不包含：登录真实实现、业务接口联调、正式统计看板内容

## 目标

- 搭建一个现代化的 Vue 3 后台管理框架，供后续页面直接挂载
- 提供路由、布局、鉴权拦截、状态管理、API 请求基础设施
- 建立一套适合系统设置、监控端、二维码、订单、API 文档页面的轻量设计基线
- 替换当前脚手架式首页，让应用具备真实后台框架形态

## 不在本次范围内

- `TP-14` 的真实登录表单和登录请求
- 系统设置、监控端、二维码、订单、API 文档的真实业务逻辑
- 图表较重的正式 Dashboard
- 引入重量级第三方 UI 组件库

## 设计方向

### 产品气质

后台整体采用现代 SaaS 仪表盘气质：

- 以浅色体系为主
- 使用带层次的蓝灰色背景氛围
- 内容区域以清晰卡片结构组织
- 导航、内容、状态信息之间有明确对比层级
- 悬停、按下、切换时加入轻量动态反馈

目标是比传统灰色后台更现代，但仍然适合高频配置和后台操作场景。

### 视觉规则

- 主内容区优先保证高可读性和明亮界面
- 品牌强调色统一收敛到蓝色 / 青蓝色系
- 优先使用细边框与柔和阴影，不做过重的玻璃拟态
- 卡片、面板、输入框、弹层保持统一圆角体系
- 保证后续表格、表单页可以自然接入，不需要推翻重做

### 响应式规则

- 本阶段以桌面端、笔记本端为主要目标
- 在较窄宽度下支持侧边栏折叠
- 内容容器在常见笔记本宽度下保持良好阅读体验
- 不为了移动端牺牲后台桌面操作效率

## 信息架构

即使业务页当前还是占位页，后台框架也应先把稳定导航结构搭好。

### 路由分组

- 公开路由
  - `/login`：为后续 `TP-14` 预留占位页
- 受保护后台路由
  - `/`
  - `/dashboard`
  - `/settings`
  - `/monitor`
  - `/qrcodes`
  - `/orders`
  - `/docs`

### 导航结构

- 总览
  - Dashboard
- 运营与配置
  - 系统设置
  - 监控端
  - 二维码管理
  - 订单管理
- 资源
  - API 文档

该结构与后续 `TP-15` 到 `TP-19` 的页面任务保持一致。

## 技术架构

### 前端目录结构

前端应整理为职责清晰的层次：

- `src/layouts`
  - 后台主布局
- `src/router`
  - 路由定义与导航守卫
- `src/views`
  - 各后台页面占位页
- `src/stores`
  - 全局状态与登录态状态
- `src/services`
  - API 客户端与请求工具
- `src/components`
  - 可复用后台基础组件
- `src/styles`
  - 设计令牌、全局样式、通用样式层

### 布局模型

受保护区域统一使用一个 `AdminLayout`：

- 左侧为主导航侧边栏
- 顶部为页面标题栏、上下文说明区和后续操作区
- 中间为统一内容容器
- 页面内部共用统一的标题区、面板区、空状态结构

### 路由守卫模型

路由系统应支持：

- 通过路由元信息标记是否需要鉴权
- 统一的前置守卫逻辑
- 未登录访问受保护路由时跳转到 `/login`
- 已登录访问 `/login` 时跳转到 `/dashboard`

在 `TP-13` 阶段，登录态可先通过 store 和本地存储标记进行演示，不要求真实登录流程。

### 状态模型

初期使用两个 store 即可：

- `auth`
  - 登录状态标记
  - 从本地存储恢复状态
  - 清理会话辅助方法
- `app`
  - 侧边栏折叠状态
  - 页面级元信息
  - 轻量全局 UI 状态

这样既够用，也便于后续扩展。

### API 层

请求层应提供：

- 从环境变量读取基础地址
- 统一 JSON 请求封装
- 统一错误格式处理
- 为后续未登录失效场景预留处理入口

本阶段重点不是联调，而是先把模式和接入点搭好。

## 组件规划

后台壳子先提供一组结构型基础组件，供后续业务页直接复用：

- `AdminSidebar`
- `AdminTopbar`
- `PageShell`
- `PageHeader`
- `InfoCard`
- `PanelCard`
- `EmptyState`
- `TableShell`
- `FormSection`

这些组件只解决结构和视觉复用，不承担具体业务逻辑。

## 页面策略

每个受保护页面都应渲染“有信息量的占位内容”，而不是空白页。

每个占位页至少包含：

- 明确的页面标题
- 页面用途说明
- 一个或多个面板容器
- 后续功能清单或接入提示

这样既能保证后台框架看起来完整，也能降低后续页面替换成本。

## 样式策略

### 设计令牌

全局 CSS 应定义：

- 语义化背景色
- 面板与边框颜色
- 文本层级颜色
- 强调色
- 圆角尺度
- 阴影尺度
- 间距节奏
- 动效时长

### 动效规则

动效保持轻量且有意义：

- 卡片和导航项提供轻微悬停抬升感
- 交互控件提供柔和的颜色与背景过渡
- 内容容器可带轻量进入动效

所有动效都应克制，保证后台仍然专业稳定。

## 可用性与体验约束

- 所有主要文本保持足够对比度
- 点击区域尺寸舒适，不做过小交互点
- 保留清晰可见的焦点态
- 选中态不能只靠颜色表达
- 占位页必须有说明，不做空白内容
- 标题、说明、操作区的视觉层级必须统一

## TP-13 实施落点

### 需要新增或重构的能力

- 替换当前单页脚手架式路由
- 增加后台布局与受保护路由树
- 增加后台页面框架类通用组件
- 增加 `auth` / `app` 两个全局 store
- 增加统一请求服务工具
- 用后台壳子内的 Dashboard 占位页替换当前简单首页

### 推荐实施顺序

1. 重构路由与路由元信息
2. 增加 store 与登录态初始化逻辑
3. 搭建后台布局与导航
4. 增加可复用结构组件
5. 创建各业务页面占位页
6. 刷新全局样式与设计令牌
7. 验证路由守卫与布局行为

## 验收标准

当以下条件满足时，可认为 `TP-13` 完成：

- 应用启动后进入完整的后台框架形态
- 受保护页面统一在后台布局中渲染
- 未登录访问受保护页面会被拦截
- 所有占位页可以通过导航正常切换
- 页面视觉已明显脱离默认脚手架样式
- 该框架可以直接承接 `TP-14` 到 `TP-19`，无需结构性返工

## 风险与应对

### 风险：业务页尚未实现时，把 UI 壳子做得太重

应对：

- 组件保持结构型和通用型
- 避免过早抽象出强业务含义组件

### 风险：视觉过度装饰，不利于后台高频操作

应对：

- 优先保证清晰度、信息承载能力和可读性
- 控制强调色和装饰元素使用强度

### 风险：`TP-13` 的伪登录态容易误导后续实现

应对：

- 明确当前登录态机制只是过渡方案
- 保持守卫逻辑容易替换为真实会话机制

## 当前假设

- 本任务阶段继续只依赖 Vue 3、Pinia、Vue Router
- 样式层继续使用自定义 CSS，不额外引入组件库
- 只要能演示路由拦截，使用占位登录态就可以满足 `TP-13`

## 结论

本设计将 `TP-13` 严格收敛为“现代化后台壳子建设”：搭出具备真实产品形态的后台框架、统一视觉基线与可复用前端基础设施，同时明确把真实登录、业务逻辑和正式页面内容留到后续任务包实现。
