# TP-17 前端二维码管理页设计
## 1. 背景

当前后台前端已经完成统一框架、登录鉴权、系统设置页和监控页，但二维码管理页仍然是占位状态，无法承接真实的固定金额二维码维护流程。现有 Node API 已经提供以下兼容接口：

1. `/admin/getPayQrcodes`
2. `/admin/addPayQrcode`
3. `/admin/delPayQrcode`
4. `/deQrcode2`
5. `/enQrcode`

同时，原始 VMQ 后台已经验证过“上传二维码图片 -> 识别内容 -> 录入金额 -> 保存固定金额二维码”的工作模式，因此本次 `TP-17` 的目标不是重新设计业务流程，而是将该能力以现代化双分区页面的方式迁移到当前前端后台。

本次设计遵循以下原则：

1. 页面必须面向生产使用，不保留任务编号、占位说明、开发指导文案。
2. 页面交互优先贴近原后台已验证的管理路径，降低迁移后的学习成本。
3. 单页内同时完成“草稿导入”和“已保存管理”，避免用户在多个页面之间来回跳转。
4. 保持与当前后台 UI 风格、日志规范和错误处理方式一致。

## 2. 目标

二维码管理页需要同时满足以下目标：

1. 管理员能够分别维护微信固定金额二维码和支付宝固定金额二维码。
2. 管理员能够一次上传多张二维码图片，系统逐张解析二维码内容。
3. 管理员能够为每条解析结果录入金额，并批量保存到后台。
4. 管理员能够查看已保存的固定金额二维码列表。
5. 管理员能够删除不再使用的固定金额二维码。

## 3. 后端能力与数据约束

### 3.1 已有接口

当前 Node API 已具备以下能力：

1. `/admin/getPayQrcodes`
   用于分页读取固定金额二维码列表，支持按 `type` 区分支付方式。
2. `/admin/addPayQrcode`
   用于新增固定金额二维码，核心字段为：
   - `type`
   - `payUrl`
   - `price`
3. `/admin/delPayQrcode`
   用于按 `id` 删除固定金额二维码。
4. `/deQrcode2`
   用于上传二维码图片并解析二维码文本内容。
5. `/enQrcode`
   用于根据二维码文本生成预览图片地址。

### 3.2 支付方式映射

页面内部统一使用以下映射规则：

1. 微信：`type = 1`
2. 支付宝：`type = 2`

该映射与原始项目和当前 Node API 保持一致，不额外引入前端枚举转换歧义。

### 3.3 数据对象

页面会维护两类前端对象：

1. 草稿对象
   用于承接用户上传但尚未保存的二维码记录，至少包含：
   - `draftId`
   - `fileName`
   - `type`
   - `price`
   - `payUrl`
   - `previewUrl`
   - `status`
   - `errorMessage`
2. 已保存对象
   直接映射后端列表返回项，至少关注：
   - `id`
   - `type`
   - `price`
   - `payUrl`
   - `createDate` 或后端等价字段

## 4. 页面结构

二维码管理页采用“单页双分区”的结构，页面只保留一个路由入口，但在内容区固定拆成两个业务区块：

1. 微信固定金额二维码
2. 支付宝固定金额二维码

每个区块内部都包含以下 3 个子区域。

### 4.1 批量上传区

用于选择多张二维码图片并展示当前批次处理状态，包含：

1. 多文件上传入口
2. 批量上传说明
3. 批量解析状态提示
4. 批量保存按钮
5. 草稿清空按钮

### 4.2 待保存草稿区

用于展示当前分区下已上传但尚未保存的二维码草稿表格。每一行需要展示：

1. 文件名
2. 二维码预览
3. 二维码内容摘要
4. 可编辑金额输入框
5. 当前状态
6. 行级移除按钮

### 4.3 已保存列表区

用于展示当前分区下已经入库的固定金额二维码记录。每一行需要展示：

1. 二维码预览
2. 金额
3. 二维码内容摘要
4. 创建时间或可用时间字段
5. 删除按钮

已保存列表提供简单分页能力，不引入复杂筛选器。

## 5. 交互流程设计

### 5.1 页面初始化

页面进入时同时发起两组列表查询：

1. 微信二维码列表
2. 支付宝二维码列表

两个分区各自维护独立的加载状态和错误提示，避免一个分区失败后阻塞另一个分区展示。

### 5.2 批量上传与解析

当用户在某个分区选择多张图片后，页面按文件顺序创建草稿项，并进入以下流程：

1. 校验文件类型是否为图片。
2. 合法文件写入草稿列表，初始状态为“待解析”。
3. 逐张调用 `/deQrcode2` 解析二维码内容。
4. 解析成功后回填 `payUrl`，并基于 `/enQrcode` 生成 `previewUrl`。
5. 解析失败时，将该草稿标记为失败，并保留文件名与错误信息。

页面允许同一批次出现“部分成功、部分失败”，不会因为单张失败中断整个批次。

### 5.3 金额录入与校验

草稿记录中的金额由用户手动输入，校验规则如下：

1. 必填。
2. 必须为合法数字。
3. 必须大于 `0`。

校验失败的记录不能参与批量保存，但仍保留在草稿区，便于用户修正后重试。

### 5.4 批量保存

点击“批量保存”后，页面只提交当前分区下满足以下条件的草稿：

1. 解析成功
2. 金额合法
3. `payUrl` 非空

保存策略采用“逐条串行提交”，每条草稿单独调用 `/admin/addPayQrcode`。这样可以：

1. 明确知道哪一条保存失败。
2. 在日志中准确记录失败上下文。
3. 与当前项目偏重排障的实现风格保持一致。

保存完成后：

1. 成功项从草稿区移除。
2. 失败项保留并展示错误提示。
3. 当前分区刷新已保存列表。
4. 页面给出成功数、失败数的汇总反馈。

### 5.5 删除已保存二维码

点击删除按钮后，页面先进行二次确认，确认后调用 `/admin/delPayQrcode`。删除成功后刷新当前分区列表；删除失败时保留原列表状态，并展示错误信息。

## 6. 展示与交互规则

### 6.1 草稿状态

草稿状态建议统一为以下几类：

1. `待解析`
2. `解析中`
3. `解析成功`
4. `解析失败`
5. `保存中`
6. `保存失败`

页面通过状态标签区分当前进度，避免用户误判某条记录是否可保存。

### 6.2 二维码内容展示

二维码内容可能较长，因此页面采用“摘要展示 + 完整内容保留”的策略：

1. 表格内只展示截断摘要。
2. 鼠标悬停或辅助说明可查看完整内容。
3. 不主动隐藏真实内容，便于管理员排查导入错误。

### 6.3 列表分页

已保存列表先采用固定页大小和页码切换，不额外引入搜索、金额筛选或排序器。分页状态按分区独立维护。

### 6.4 错误提示

错误提示遵循“就近展示”的规则：

1. 列表加载失败：显示在当前分区列表区域。
2. 单张解析失败：显示在对应草稿行。
3. 批量保存失败：显示在分区批量操作区域，同时保留失败草稿。
4. 删除失败：显示在当前分区列表区域。

## 7. 前端实现拆分

计划新增或改造以下前端层次：

1. `QrcodeView.vue`
   承接整页布局、双分区状态管理、上传交互、草稿编排和列表刷新。
2. `qrcode-admin-service.js`
   封装二维码管理页所需的真实接口调用，包括：
   - 列表查询
   - 新增二维码
   - 删除二维码
   - 批量解析辅助
   - 预览地址生成
3. `qrcode-admin.js` 或等价工具文件
   封装纯函数逻辑，例如：
   - 支付方式映射
   - 草稿对象创建
   - 金额校验
   - 文本摘要格式化
4. 页面测试文件
   覆盖服务层参数映射和页面核心交互行为。

若页面脚本过大，可在保持目录清晰的前提下抽取局部组件，但不为了拆分而拆分。

## 8. 日志要求

根据当前项目规则，关键逻辑、关键流程和关键异常分支必须打印日志。本页至少记录以下节点：

1. 某分区开始批量解析图片
2. 单张图片解析成功
3. 单张图片解析失败
4. 某分区开始批量保存
5. 单条二维码保存失败
6. 某分区批量保存结束及汇总结果
7. 删除二维码成功
8. 删除二维码失败

日志使用中文说明，并带上必要上下文，例如支付方式、文件名、记录 id、失败原因等。

## 9. 测试方案

前端至少覆盖以下测试场景：

1. 页面初始化时双分区独立加载列表。
2. 多图上传后生成草稿，并正确处理解析成功与失败分支。
3. 金额校验生效，非法金额不能进入保存流程。
4. 批量保存后成功项移除、失败项保留，并刷新已保存列表。
5. 已保存列表删除成功后刷新，删除失败时保留原状态。
6. 服务层正确映射 `type`、分页参数、删除 `id` 和新增参数。

## 10. 验收标准

满足以下条件即可视为 `TP-17` 完成：

1. 二维码管理页不再是占位页。
2. 页面以双分区方式分别管理微信和支付宝固定金额二维码。
3. 页面支持一次上传多张二维码图片。
4. 页面能够逐张解析二维码内容，并展示成功或失败状态。
5. 页面允许用户录入金额并批量保存解析成功的二维码。
6. 页面能够展示已保存的固定金额二维码列表。
7. 页面能够删除已保存二维码。
8. 页面不保留任务编号、占位提示或开发说明文案。
9. 页面日志、注释和错误提示满足当前项目规范。

## 11. 本次范围外

以下内容不纳入本次 `TP-17`：

1. 二维码列表高级筛选和搜索。
2. 拖拽排序或批量删除。
3. 金额重复性的前端强校验拦截。
4. 导入历史回溯。
5. 跨分区批量混合保存。
