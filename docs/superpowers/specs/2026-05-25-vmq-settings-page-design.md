# TP-15 前端系统设置页设计说明

## 背景

`TP-15` 的目标是在现有后台框架与登录鉴权闭环已经完成的基础上，落地真实可用的系统设置页。

当前已知前提如下：

- `TP-13` 已提供稳定的后台壳子、页面结构组件和统一视觉风格
- `TP-14` 已接入真实登录与基于后端 session 的前端鉴权
- 后端已提供 `/admin/getSettings`、`/admin/saveSetting`、`/deQrcode`、`/deQrcode2`、`/enQrcode` 等接口
- 本次要求保持既有后台风格，不新增新的视觉体系
- 面向生产的页面中不得保留开发指导性质文案

## 目标

- 将系统设置页从占位页升级为真实业务页
- 接通系统设置的读取、编辑与保存
- 支持二维码文本录入、图片上传解析与预览
- 将微信和支付宝配置区域分开管理
- 在保存和解析操作中提供明确且克制的成功/失败反馈

## 不在本次范围内

- 修改后端设置项定义
- 增加新的后端设置接口
- 调整后台整体布局结构
- 增加复杂权限粒度控制
- 增加批量导入导出、历史版本或审计日志能力

## 设计方向

### 页面结构

系统设置页采用单页分区式工作台结构，继续使用后台既有的：

- `PageShell`
- `PageHeader`
- `PanelCard`
- `FormSection`

页面主体分为三块：

1. 基础配置区
2. 支付二维码配置区
3. 预览与保存区

### 视觉原则

- 延续现有浅色 SaaS 后台基线
- 保持卡片式分区和一致的输入控件样式
- 交互反馈清晰但不过度打扰
- 页面只展示真实业务信息，不保留任务编号、占位说明等开发文案

## 信息架构

### 基础配置区

基础配置区至少承接以下可编辑字段：

- 后台账号 `user`
- 后台密码 `pass`
- 异步通知地址 `notifyUrl`
- 同步返回地址 `returnUrl`
- 商户 key `key`
- 订单超时关闭时间 `close`
- 金额区分模式 `payQf`

### 支付二维码配置区

二维码配置区按支付方式拆分为两块：

- 微信二维码配置 `wxpay`
- 支付宝二维码配置 `zfbpay`

每块都支持两种输入方式：

1. 直接输入或粘贴二维码原始文本
2. 上传二维码图片并解析

### 预览与保存区

预览与保存区负责：

- 展示当前微信二维码预览
- 展示当前支付宝二维码预览
- 展示解析摘要
- 提供统一保存按钮
- 展示保存结果反馈

## 技术架构

### 前端服务层

建议新增系统设置服务文件，统一封装：

- `fetchSettings()`
- `saveSettings(payload)`
- `decodeQrcodeFromText(base64)`
- `decodeQrcodeFromFile(file)`
- `buildQrcodePreviewUrl(text)`

这样可以将接口细节与页面逻辑拆开，降低页面体积。

### 表单辅助层

建议新增轻量表单辅助模块，负责：

- 生成默认表单值
- 将接口返回值映射为表单结构
- 将表单结构映射为保存 payload
- 基础前端校验

### 页面状态

系统设置页需要至少维护以下状态：

- `loading`
- `saving`
- `saveMessage`
- `saveError`
- `form`
- `wechatPreviewUrl`
- `alipayPreviewUrl`
- `wechatDecodeMessage`
- `alipayDecodeMessage`

## 交互设计

### 进入页面

页面初始化时：

- 先请求系统设置接口
- 成功后回填表单
- 同时根据已有二维码文本生成预览
- 失败时在页面内联提示加载失败

### 文本二维码处理

用户修改二维码文本后：

- 可立即更新预览
- 如文本为空，则清空对应预览
- 不强制每次输入都请求后端解析

### 图片二维码处理

用户上传二维码图片后：

- 调用上传解析接口
- 解析成功后写回对应二维码文本字段
- 同步刷新预览
- 解析失败时只在当前区块提示错误，不影响其他输入

### 保存行为

点击保存后：

- 按钮进入 loading 状态
- 页面不清空已输入内容
- 保存成功后显示成功反馈
- 保存失败后显示明确错误反馈

## 校验规则

前端只做最小必要校验：

- 后台账号不能为空
- 后台密码不能为空
- `close` 必须是非负整数或数字字符串
- URL 字段若填写，则做基础格式校验

最终结果以后端响应为准，不在前端增加与后端不一致的复杂限制。

## 测试策略

### 单元测试

优先覆盖以下可纯逻辑验证部分：

- 表单默认值和映射逻辑
- 保存 payload 生成逻辑
- 基础校验逻辑
- 二维码服务层请求封装

### 集成验证

至少验证以下场景：

- 页面可读取并回填后端设置
- 修改后可保存成功
- 二维码文本可生成预览
- 上传二维码图片后可解析并回填
- 微信与支付宝区域互不干扰

## 验收标准

满足以下条件即可认为 `TP-15` 完成：

1. 系统设置页能真实读取后端配置
2. 修改后可真实保存，并再次读到一致结果
3. 微信和支付宝二维码可分别通过文本或图片上传方式处理
4. 页面能展示二维码预览或解析摘要
5. 页面风格与当前后台保持一致，且不含开发指导性文案

## 风险与应对

### 风险：页面逻辑过于集中，导致文件膨胀

应对：

- 服务层、表单映射与页面展示职责分离
- 页面只负责状态编排与事件触发

### 风险：二维码解析失败干扰整页使用

应对：

- 将错误提示局限在当前二维码区块
- 解析失败不清空其他字段

### 风险：保存 payload 与后端字段名不一致

应对：

- 统一通过表单辅助模块做双向映射
- 使用测试锁定字段映射

## 当前假设

- 后端 `getSettings` 返回的 `data` 是以设置 key 为键的对象
- 后端 `saveSetting` 接口接受与当前设置 key 同名的字段
- 二维码文本预览可以通过 `/enQrcode?url=` 方式生成

## 结论

本设计将 `TP-15` 收敛为“真实系统设置表单 + 二维码解析与预览 + 设置保存闭环”：不扩展后端协议，不改变后台风格，只把前端系统设置页从占位态升级为可投入实际使用的生产页面。
