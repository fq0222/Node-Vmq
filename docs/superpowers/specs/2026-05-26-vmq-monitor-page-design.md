# TP-16 前端监控端页面设计

## 1. 背景

当前后台框架已经完成登录、系统设置和统一布局，但监控端页面仍是占位页，无法承载真实监控端接入流程。后端已经提供监控状态查询能力，旧项目也已有监控端绑定和 APK 下载的完整使用路径，因此本次 `TP-16` 目标是将监控端页面升级为真实业务页，直接对接现有后端能力。

本次设计遵循以下原则：

1. 保持与当前后台一致的轻量 SaaS 风格。
2. 面向真实使用，不保留任务编号、占位说明或开发指导文案。
3. 优先打通核心链路：状态查看、绑定接入、APK 下载。

## 2. 目标

监控端页面需要同时承载“状态查看”和“接入配置”两类任务：

1. 管理员能够查看监控端当前是否在线。
2. 管理员能够看到最近一次心跳时间和最近一次收款时间。
3. 管理员能够通过扫码或手动配置两种方式接入监控端。
4. 管理员能够直接获取监控端 APK 下载入口。

## 3. 现有后端能力

### 3.1 状态接口

后端监控端兼容服务已提供 `/getState`，返回值包含：

1. `state`
2. `lastheart`
3. `lastpay`

状态值语义：

1. `1`：在线
2. `0`：离线
3. `-1`：未绑定

### 3.2 配置数据来源

系统设置中已有以下与监控端相关的配置源：

1. 通讯密钥 `key`
2. 异步通知地址 `notifyUrl`
3. 同步返回地址 `returnUrl`

本次前端会至少读取 `key` 作为绑定配置核心参数，并结合当前 API 基础地址生成绑定串和二维码内容。

### 3.3 APK 入口来源

旧项目存在监控端 APK 文件 `v.apk`，前端页面会直接提供真实下载入口，避免只给文案不给操作。

## 4. 页面结构

监控端页面采用“单页工作台”结构，分成 4 个区域。

### 4.1 实时状态区

显示 3 个状态卡片：

1. 当前状态
2. 最后心跳时间
3. 最后收款时间

同时放置“刷新状态”按钮，便于调试和运营人员手动确认当前状态变化。

### 4.2 扫码绑定区

显示：

1. 绑定二维码
2. 绑定串原文
3. 复制绑定串按钮

二维码内容和绑定串内容来自同一份配置字符串，避免扫码结果和手动配置不一致。

### 4.3 手动配置区

按字段方式展示监控端接入所需的关键信息，至少包含：

1. 服务端地址
2. 通讯密钥
3. 状态查询地址
4. 心跳上报地址

这一区域的目标不是解释实现原理，而是让管理员可以直接照着填。

### 4.4 APK 与检查区

提供：

1. APK 下载按钮
2. 安装后检查项列表

检查项只保留真实使用需要的最小内容，例如：

1. 安装应用
2. 开启辅助服务
3. 完成绑定后刷新状态

## 5. 数据流设计

### 5.1 页面初始化

页面加载时并行执行两类请求：

1. 读取系统设置
2. 读取监控状态

系统设置成功后，前端生成：

1. 绑定串
2. 二维码预览地址
3. 手动配置展示字段

监控状态成功后，前端更新状态卡片。

### 5.2 手动刷新

点击“刷新状态”后只重新请求监控状态，不重新加载整页配置，缩短等待时间。

### 5.3 二维码与绑定串生成

前端基于以下内容生成绑定配置：

1. API 服务基础地址
2. 通讯密钥
3. 监控端关键接口地址

绑定串是扫码配置和复制配置的唯一来源。二维码只是绑定串的图形化表达。

## 6. 展示与交互规则

### 6.1 状态映射

状态值按以下规则转换：

1. `1` -> `在线`
2. `0` -> `离线`
3. `-1` -> `未绑定`
4. 其他值 -> `未知状态`

同时配合不同颜色的状态标签，增强辨识度。

### 6.2 时间格式化

`lastheart` 和 `lastpay` 为 13 位时间戳时，格式化为本地可读时间。

当值为空、`0` 或非法时：

1. 页面显示 `暂无记录`
2. 不展示错误弹窗

### 6.3 复制反馈

点击复制绑定串后：

1. 成功时显示简洁成功提示
2. 失败时显示简洁错误提示

不跳转页面，不清空现有内容。

### 6.4 错误处理

1. 状态请求失败时，仅在状态区显示页内错误提示
2. 设置请求失败时，绑定区和手动配置区显示页内错误提示
3. 某一区域失败不应阻塞另一块已成功的数据展示

## 7. 前端实现拆分

计划新增或改造如下前端层次：

1. `MonitorView.vue`：承载整页布局与交互
2. `monitor-service.js`：封装监控状态读取、监控配置生成逻辑
3. `monitor-format.js`：处理状态映射、时间格式化、绑定串生成等纯函数
4. 监控页测试文件：覆盖核心数据转换和接口调用行为

如页面结构较大，可在不破坏目录简洁度的前提下，抽出局部小组件，但不做过度拆分。

## 8. 验收标准

满足以下条件即可视为 `TP-16` 完成：

1. 监控端页面不再是占位页。
2. 页面能真实读取并展示监控端状态。
3. 状态文案与颜色能正确反映在线、离线、未绑定。
4. 页面能展示扫码绑定二维码。
5. 页面能展示并复制绑定串。
6. 页面能展示手动配置所需信息。
7. 页面能提供真实 APK 下载入口。
8. 页面整体风格与当前后台一致。
9. 页面中不保留任务编号、开发提示或占位性质文案。

## 9. 风险与约束

1. `/getState` 为监控端兼容接口，签名规则沿用旧版协议，前端如需直接请求该接口，需要保证签名和时间戳生成逻辑一致。
2. 若后端未额外提供“后台内部查询监控状态”的免签接口，则前端服务层需要在当前系统配置基础上完成签名参数拼装。
3. APK 文件路径必须以当前实际可访问地址为准，避免前端展示了无效下载链接。

## 10. 本次范围外

以下内容不纳入 `TP-16`：

1. 监控端 App 本体改造
2. 自动轮询刷新
3. 复杂的接入向导
4. 监控端异常日志查看
