# Vmq Node.js + Vue 3 + PostgreSQL 重构执行计划

## 1. 计划目标

本计划用于指导将原 Vmq 服务端与 Web 管理后台重构为 Node.js + Express、Vue 3 + Vite、PostgreSQL 的实施过程。

执行原则：

- 优先保证业务逻辑不变
- 优先保证接口兼容
- 优先保证监控端可继续对接
- 采用分阶段交付，降低一次性替换风险

## 2. 执行策略

推荐采用以下总体策略：

- 第一优先级：梳理并冻结兼容协议
- 第二优先级：先重建后端领域逻辑与数据库模型
- 第三优先级：在后端稳定后重建前端管理后台与支付页
- 第四优先级：完成联调、回归、迁移与上线准备

这是一次“保守型重构”，不追求一次性架构炫技，而追求逻辑对齐、接口稳定和可分批验收。

## 3. 拆分原则

模块拆分按“业务闭环”而非单纯技术层进行：

- 一个阶段尽量形成可验证的业务能力
- 每个模块都有明确输入、输出、依赖、验证方式
- 高风险模块优先独立验证

### 目标

3. 数据库与初始化
4. 共享规则

- 再做高风险业务链路
- 最后做界面与上线动作

## 7. 风险控制策略

## 7.1 高风险点

- 并发创建订单导致金额占位错乱
- `price/reallyPrice` 精度问题
- PostgreSQL 数值类型与旧逻辑差异
- 监控端签名与时间戳兼容问题
- 回调成功判定误差
- 前后端分离后会话失效问题

## 7.2 控制措施

- 金额逻辑先写测试再实现
- 所有签名样例做固定回归样本
- 关键接口保留兼容层
- 在联调前制作 Postman/Apifox 回归集合
- 用旧项目真实样例做对照测试

## 8. 验收标准

判定本次重构可进入上线准备，至少要满足：

- 对外核心接口兼容通过
- 安卓监控端联调通过
- 创建订单到收款回调全链路通过
- 超时关闭与监控离线逻辑通过
- 后台全部核心功能可用
- 支付页完整链路可用
- 主要回归场景全部通过

## 10. 本计划结论

这个项目最合适的落地方式不是“边写边改”，而是：

- 先冻结兼容要求
- 再围绕高风险业务链路逐层重建
- 用回归测试证明逻辑未变

这样后续不管是单人推进还是多人拆分，都比较稳，也方便你把任务继续拆到接口级、页面级和表级。
