# 项目实施准则

## 代码规范

1. 项目中的代码必须编写注释。
2. 注释必须使用中文。
3. 所有 `.js` 文件都必须在文件头使用整体注释说明该文件作用。
4. 关键逻辑、关键流程、关键异常分支必须打印日志，便于排查问题。
5. 面向生产使用的页面中，不得保留用于开发指导、阶段标记、任务编号、占位说明等提示性文案。

## 提交规范

1. 代码提交必须由用户发起。
2. 未经用户明确要求，助手不得主动创建提交。
3. 所有 Git Commit 信息必须使用中文。
4. Git Commit 信息必须使用以下类型前缀：
   - `feat`：新增功能
   - `fix`：修复缺陷
   - `docs`：文档更新，如 `README.md`、API 文档、设计说明等
   - `refactor`：代码重构，既不是新增功能也不是修复缺陷
   - `style`：代码格式调整，如空格、分号、缩进等，不影响代码逻辑
   - `test`：添加或修改测试用例
   - `chore`：构建过程或辅助工具的变动，如脚本、规范、工具配置等
   - `perf`：性能优化
   - `ci`：持续集成相关配置文件或脚本修改
   - `build`：编译相关变动，如构建系统、外部依赖包等
   - `revert`：撤销以前的提交
5. Commit 信息格式统一为：`类型: 中文描述`
6. 使用 Git 工具执行提交相关操作时，必须单条命令串行执行，不得并发调用多条 Git 命令，避免触发锁文件冲突。
